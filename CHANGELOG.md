- «AI-анализ» / «Уверенность AI»: честные и математически согласованные показатели совпадений, тултипы к каждой карточке, процент как основное значение.
  - **Что.**
    1. **Единый знаменатель и согласованность.** Раньше «Принято без изменений»/«Исправлено» считались от ВСЕХ AI-оценок (`total`, включая непроверенные и без AI-снимка), а «Точное совпадение»/«Несовпадение» — от `comparable` (проверенные с AI-снимком). При этом «принято ⇔ точное совпадение» и «исправлено ⇔ несовпадение» описывают ОДНИ И ТЕ ЖЕ записи, поэтому одинаковые по смыслу показатели давали разные проценты и одинаковые абсолютные числа в разных карточках — отсюда ощущение «повторяющихся/несогласованных цифр». Теперь все проценты совпадения считаются от ЕДИНОГО множества «валидные проверки» (`validReviewed` = проверено + есть исходный AI-снимок), а «Принято/Исправлено» — от `reviewed`.
    2. **exact% + mismatch% = 100%.** Основная карточка «Несовпадение» показывает ШИРОКОЕ несовпадение (= не точное совпадение = частичное + значимое), `mismatchBroadPct = 100 − exactPct`. «Частичное совпадение» теперь явно ПОДМНОЖЕСТВО несовпадений (а не третья взаимоисключающая категория) с долей совпавших критериев (`совпавшие критерии ÷ все сравнимые критерии`).
    3. **«Недостаточно данных» — отдельная карточка.** Проверенные строки без исходного AI-снимка показываются отдельно и ИСКЛЮЧЕНЫ из знаменателя (не считаются ни совпадением, ни несовпадением).
    4. **Процент — основное значение, абсолют ниже** (требование менеджера): «70%» крупно, «84 из 120 проверок» ниже — для точного совпадения, несовпадения, частичного, принято/исправлено.
    5. **Тултипы к каждой карточке + блок-инструкция** «Как читать показатели»: что означает, какие поля сравниваются, формула, знаменатель, исключаются ли неполные записи, пересечение категорий (RU, бизнес-язык).
  - **Зачем.** Обратная связь менеджера: «нужны тултипы/инструкция — что такое точное совпадение, сколько и каких процентов должно совпасть, и в остальных так же; и в точном совпадении, и в несовпадении основным показателем сделать %, а ниже — абсолютное число». Плюс аудит: показатели «повторяли одни и те же числа» из-за разных знаменателей для одного и того же множества записей.
  - **Как.** Только слой аналитики и UI, БЕЗ изменения схемы БД и логики сохранения (исходный AI-снимок и уверенность уже фиксируются один раз и не перезаписываются — проверено в `repo.ts`/`confidence-repo.test.ts`). `src/lib/match.ts` — добавлены `fieldsTotal`/`fieldsMatched` (согласие по критериям); `src/lib/confidence-report.ts` — `matches.validReviewed/mismatchBroad/mismatchBroadPct/partialPct/partialFieldsAgreementPct` + `acceptedOfReviewedPct` (аддитивно, старые поля сохранены ради совместимости); `src/components/ConfidenceReport.tsx` — тултипы (`InfoDot`), блок-инструкция, `%`-первичная вёрстка карточек.
  - **Результат.** `npm test` — 557 зелёных (новый файл `tests/confidence-consistency.test.ts` — 12 тестов по обязательным кейсам ТЗ: exact%+mismatch%=100, «Недостаточно данных» вне знаменателя, частичное как подмножество, «принято» не выводится из уверенности, низкая уверенность/принято и высокая/исправлено; плюс 2 теста счётчиков полей в `tests/match.test.ts`); `tsc --noEmit` по `src/` — без ошибок; `npm run build` — успешно. Что: (1) первопричина — фича «Уверенность AI» (PR #18) начала писать в `mqa_evaluations` колонки `ai_confidence`/`ai_total`/`review_status`/`reviewed_by`/`reviewed_at`, но миграция `20260716_mqa_evaluation_confidence.sql` не была применена к боевой БД, поэтому КАЖДОЕ сохранение оценки падало; миграция применена к боевому проекту — сохранение снова работает (перезапуск деплоя не требуется, чинилась схема БД, не код). (2) Почему на экране было «[object Object]»: Supabase/PostgREST бросают простой объект (`{ message, details, hint, code }`), а не `Error`, и `dbErrorResponse` делал `String(e)` → буквально «[object Object]», которое UI и показывал. Добавлен помощник `errorMessage()` (тянет `.message`/`.details`/`.hint` + код pg), `dbErrorResponse` использует его; маршруты `PUT`/`DELETE` в `evaluations/[id]` переведены на `dbErrorResponse` для единообразных сообщений и корректных кодов статуса. Зачем: Маргарита не могла сохранять оценки, а сообщение об ошибке было нечитаемым. Результат: `npm run build` успешен; типы изменённых файлов чисты; боевые колонки на месте (проверено в БД). Впредь любая ошибка БД читаема, а не опаковый «[object Object]».

- Отображение нарушений приведено к требованиям: один чат = одна строка (несколько нарушений по чату — через запятую, движок `groupNarusheniya`); сотрудники без нарушений не выводятся (п.9), общий пустой статус «Нет нарушений за выбранный период»; в расчёты попадают только подтверждённые ручные нарушения (`confirmed !== false`) — в дашборде, сообщениях и журнале. Миграция `20260714_mqa_exclude_auto_from_score_violations.sql` пометила 38 исторических авто-нарушений (`note='авто из оценки'`) как `confirmed=false` (исключены из расчётов, строки не удалены). Применена к боевой БД. Осталось 64 ручных нарушения.
