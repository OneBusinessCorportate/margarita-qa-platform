# Журнал изменений

Запись формата «что / зачем / результат» обязательна для каждого PR, меняющего код: без неё робот `doc-sync` краснит и мердж заблокирован. Форточка для заведомо тривиальной правки — метка PR `skip-changelog`.

## 2026-07-14
- Заведён журнал изменений, коммит-gate (`doc-sync`) и защита ветки `main`. Зачем: изменение кода не должно попадать в продукт без следа «что поменяли». Результат: работа только через PR, мердж на красном физически заблокирован для всех.
- Убрана QA-концепция «Без ответа» полностью: страница `/unanswered` → редирект на `/dashboard`, ссылка из навигации, секции в отчёте/PDF/сообщениях, поля `unansweredChats` в модели отчёта, API `/api/unanswered/*`, `lib/unanswered.ts` и связанные функции в `repo.ts`, тесты. Зачем: функциональность признана лишней и путала бухгалтеров. Результат: исходные Telegram-сообщения в БД не тронуты, отчёты/PDF/сообщения больше не считают и не показывают «Без ответа»; поле `mqa_chats.unanswered` сохранено (используется только для порядка списка на странице оценки).
- Удалена страница «Регистрация» (`/registration`, еженедельный журнал QA менеджеров) → редирект на `/dashboard`; убраны `RegistrationPanel`, API `/api/manager-evaluations`, функции журнала в `repo.ts`, типы `ManagerEvaluation`/`NewManagerEvaluationInput` и поле стора. Зачем: страница выведена из эксплуатации. Результат: схема оценки `registration` в «Регламенте» (калькуляторы) сохранена — трогали только журнал.
- Убран чекбокс «Повторяющаяся задача» при создании задачи (страница «Задачи» и модалка на «Оценке»): из UI, состояния формы, payload. Зачем: по запросу. Результат: создание/закрытие задач работает; колонка «Повтор»/«QA подтвердил» в таблице задач убрана (в проде задач ещё не было).
- Полностью удалено автоматическое создание нарушений («Авто из оценки»): убран авто-POST нарушения при оценке «Критично» (`ScoringPanel`), функция `importCriticalChatsAsViolations` и маршрут `/api/violations/import-critical`, кнопка «Импортировать критичные чаты» и бейджи «Авто-импорт/не подтв.». Зачем: нарушения вносит только Маргарита вручную. Результат: скоринг больше не создаёт нарушений; ручные нарушения и их история сохранены.
- Отображение нарушений приведено к требованиям: один чат = одна строка (несколько нарушений по чату — через запятую, движок `groupNarusheniya`); сотрудники без нарушений не выводятся (п.9), общий пустой статус «Нет нарушений за выбранный период»; в расчёты попадают только подтверждённые ручные нарушения (`confirmed !== false`) — в дашборде, сообщениях и журнале. Миграция `20260714_mqa_exclude_auto_from_score_violations.sql` пометила 38 исторических авто-нарушений (`note='авто из оценки'`) как `confirmed=false` (исключены из расчётов, строки не удалены). Применена к боевой БД. Осталось 64 ручных нарушения.
