# Шаблонные уведомления клиентам — инструкция по ролям и запуску

Переворот логики рассылок: платформа **планирует** уведомления, бухгалтер их
**видит/правит/прикладывает** (в приложении бухгалтера — repo #2
`kk-accountants-feedback-form`, страница «Уведомления»), а **бот отправляет** по
расписанию. Каждая отправка **логируется** с полным текстом. Шаблонные сообщения
уходят клиенту ТОЛЬКО ботом, вручную их не набирают.

## Роли

- **Бухгалтер.** Открывает «Уведомления», видит предстоящие сообщения своих
  клиентов с предупреждением «это БУДЕТ отправлено». Может отредактировать текст
  (правка логируется), для РУЧНЫХ типов — **приложить документ** (зарплатная
  ведомость, налоговый отчёт) или отметить «сделано», подтвердить или отменить.
  Если ничего не делать — бот отправит запланированное сообщение как есть.
  Приложенный файл **отправляется клиенту** вместе с текстом (см. «Что и как
  отправляется»).
- **Наира.** Источник фиксированного текста АВТО-шаблонов (оплата услуг/долги,
  первичка). Правки текста шаблонов — через каталог `mqa_notification_templates`.
- **Менеджер / супервайзер.** Обзор «Рассылки (день)» (repo #2,
  `/notifications-daily`) — все уведомления по дням, можно вмешаться и отменить.

## Что и как отправляется

- Скрипт `scripts/send-notifications.ts` (cron `margarita-send-notifications`,
  render.yaml) берёт из `mqa_planned_notifications` строки со сроком «сегодня и
  раньше» в статусах `planned/edited/approved`.
- Для каждой строки `sendDecision()` решает: **send / dry-run / skip**.
- При `send`: боту уходит `rendered_text`; если для РУЧНОГО типа приложен файл
  (`mqa_notification_attachments.file_url`) — он **пересылается клиенту**
  (`sendDocument` по URL, текст идёт подписью). Строка помечается `sent`, факт
  пишется в `mqa_sent_notifications` (дата, полный текст, тип, подтип, договор).

## Гейт (по умолчанию ВЫКЛ — ничего клиенту не уходит)

Живая отправка клиенту происходит только когда ВСЁ верно:

1. шаблон `mqa_notification_templates.approved = true` (владелец утвердил текст);
2. переменная окружения `NOTIFICATIONS_SEND_ENABLED = 1` (на cron-сервисе);
3. для РУЧНЫХ типов — приложен файл / отмечено «сделано».

Иначе строка идёт как **dry-run** (решение принято и залогировано в stdout, но
клиенту НИЧЕГО не отправляется). Это та же посадка «no-op до конфигурации», что у
существующих Telegram-отчётов.

## Секреты (в Render dashboard, не в git)

- `NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` — доступ к БД.
- `TELEGRAM_BOT_TOKEN` — токен бота (тот же, что для отчётов).
- `NOTIFICATIONS_SEND_ENABLED` — `1`/`true` для реальной отправки (иначе dry-run).

## Планирование (30 дней)

`mqa_plan_notifications()` (pg_cron, ежедневно 07:00 Ереван) строит/освежает
цепочку: по одной строке на активный чат на каждую категорию текущего цикла,
идемпотентно — НЕ трогает строки, которые бухгалтер уже правил/подтвердил/отменил.

## Бэклог (не сделано)

Автодетект смены языка компании по имени чата — `mqa_backlog_notes`
(`notification-language-autodetect`). Язык хранится вручную в `mqa_chats.language`.
