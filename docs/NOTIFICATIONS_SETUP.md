# Шаблонные уведомления клиентам — инструкция по ролям и структуре

Переворот логики рассылок: платформа **планирует** уведомления, бухгалтер их
**видит/правит/прикладывает** (в приложении бухгалтера — repo #2
`kk-accountants-feedback-form`, страница «Уведомления»), а **бот отправляет** по
расписанию. Каждая отправка **логируется** с полным текстом. Шаблонные сообщения
уходят клиенту ТОЛЬКО ботом, вручную их не набирают.

> Этот PR даёт СТРУКТУРУ: таблицы, каталог шаблонов, планировщик на 30 дней,
> таблицу журнала отправок и язык компании. Сам **бот-отправитель** (доставка в
> Telegram + запись в журнал) выносится в отдельный focused follow-up PR, чтобы
> его семантику доставки (идемпотентность/повторные попытки/крэш-восстановление)
> ревьюить прицельно. Живой клиентской отправки в этом PR НЕТ.

## Роли

- **Бухгалтер.** Открывает «Уведомления», видит предстоящие сообщения своих
  клиентов с предупреждением «это БУДЕТ отправлено». Может отредактировать текст
  (правка логируется), для РУЧНЫХ типов — приложить документ (зарплатная
  ведомость, налоговый отчёт) или отметить «сделано», подтвердить или отменить.
  Если ничего не делать — бот (follow-up) отправит запланированное сообщение как
  есть.
- **Наира.** Источник фиксированного текста АВТО-шаблонов (оплата услуг/долги,
  первичка). Правки текста шаблонов — через каталог `mqa_notification_templates`.
- **Менеджер / супервайзер.** Обзор «Рассылки (день)» (repo #2,
  `/notifications-daily`) — все уведомления по дням, можно вмешаться и отменить.

## Структура (этот PR)

- `mqa_notification_templates` — каталог клиентского текста по
  `(category, subtype, language)`, тег `mode` AUTO/MANUAL, флаг `approved`
  (по умолчанию `false` — владелец утверждает текст до включения бота).
- `mqa_planned_notifications` (+ `mqa_notification_edits` — аудит правок) —
  очередь на 30 дней; статусы `planned/edited/approved` подлежат отправке,
  `cancelled/sent/skipped` — нет.
- `mqa_notification_attachments` — ручные вложения (файл по месяцу / «сделано»).
- `mqa_sent_notifications` — журнал: дата, полный текст, тип, подтип, договор.
- `mqa_plan_notifications()` — строит/освежает цепочку (идемпотентно, не трогает
  изменённые строки), pg_cron ежедневно 07:00 Ереван. Функция SECURITY DEFINER,
  EXECUTE отозван у PUBLIC (запускает только владелец/pg_cron).
- `mqa_chats.language` — язык клиента (default `ru`, ручное переопределение).

## Гейт live-отправки (follow-up)

Когда бот-отправитель будет добавлен, живая клиентская отправка должна включаться
только если: шаблон `approved=true` И `NOTIFICATIONS_SEND_ENABLED=1` И (для
РУЧНЫХ типов) приложен файл / отмечено «сделано». По умолчанию — dry-run.

## Тест-режим (перед боевой отправкой) — роль «Менеджер/супервайзер»

Пока задан `NOTIFICATIONS_TEST_CHAT_ID`, НИ ОДНО сообщение не уходит реальному
клиенту — всё идёт в ОДИН тест-чат, чтобы менеджер вычитал перед боевым
включением (снять переменную → пойдёт клиентам). Что видит менеджер в тест-чате:

- **Имя компании в каждом сообщении.** Так как всё падает в один чат, каждое
  сообщение предваряется шапкой «🏢 {компания} · {договор} · {категория}» —
  видно, какому клиенту оно предназначено. Реальный клиент шапку НЕ видит.
- **Ежедневный отчёт «если есть что».** Перед сообщениями бот шлёт один дайджест
  «📋 Рассылка за {дата} — N уведомл. по M компаниям» со списком; отчёт строится
  РОВНО из набора, который будет отправлен в этом прогоне (без расхождений). В
  дни, когда ничего не наступило, бот молчит (ни отчёта, ни сообщений).
- **По сроку, как в бою.** По умолчанию тест-режим фильтрует по дате: ежедневный
  крон шлёт только реально наступившее на сегодня. Разовый демо-прогон вне срока
  — только явным `--preview` / `NOTIFICATIONS_TEST_PREVIEW=1`; `NOTIFICATIONS_TEST_LIMIT`
  ограничивает число сообщений за прогон.

## Персонализация текста (реальные данные, без выдумок)

Планировщик подставляет в шаблон только то, что есть в БД:
- `{company}` — имя клиента (`mqa_chats.name_agr` → `chat_name` → `agr_no`).
- `{amount}` — реальный долг за услуги (`mqa_chats.debts`, первое целое число).
  Отрицательное значение = переплата/кредит → напоминание НЕ отправляется.
  Спец: `parseDebtAmount`/`owesServices` в `src/lib/notifications.ts` (тест-спека
  того, что делает SQL).
- `{period}` — диапазон оплаты (напр. 24.07-24.08), `{month}` — месяц запроса
  первички (локализован RU/HY/EN), `{due_day}` — срок категории, `{hvhh}` — ИНН.

**Долг только при наличии.** Напоминание об оплате услуг планируется ТОЛЬКО при
реальном положительном долге (`debts > 0`). Клиентов без долга/с переплатой не
беспокоим.

**Суммы налогов и зарплаты в БД НЕТ** — их считает бухгалтер. Эти уведомления
MANUAL: фикс-текст + приложенный PDF расчёта налогов / таблица ЗП. Реквизиты
налогов — в приложенном PDF; реквизиты оплаты услуг (Converse Bank / Business
Tech LLC) — фиксированные, литералом в тексте шаблона. Бот НЕ выдумывает суммы и
реквизиты. Бухгалтер проверяет/правит сумму долга перед отправкой (правка
доступна в любой момент до отправки).

## Бэклог (не сделано)

Автодетект смены языка компании по имени чата — `mqa_backlog_notes`
(`notification-language-autodetect`). Язык хранится вручную в `mqa_chats.language`.
